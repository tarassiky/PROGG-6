# Лабораторная работа: Сравнительный анализ производительности REST API фреймворков
---

**Дисциплина:** Программирование-6  
**Технологии:** Flask, asyncio, aiohttp, Bootstrap  
**Выполнил:** Тарасова ДЮ, группа ИВТ - 1.2  

## 1. Цель работы

Закрепить навыки нагрузочного тестирования веб-приложений, освоить методику сравнительного анализа производительности синхронных и асинхронных фреймворков, оценить влияние блокирующих операций на пропускную способность, сформулировать рекомендации по оптимизации, а также применить LLM-ассистент GigaCode с использованием техник zero-shot, one-shot и few-shot для генерации кода и тестовых сценариев.

---

## 2. Задачи

1. Реализовать тестовое REST API на трёх фреймворках (минимум один синхронный и два асинхронных, кроме FastAPI).
2. Для каждого фреймворка создать два эндпоинта: `/cpu` (блокирующий) и `/cpu_fixed` (неблокирующий).
3. Провести нагрузочное тестирование с помощью Locust при 50, 100 и 200 одновременных пользователях.
4. Собрать метрики: RPS, средняя латентность, p95, p99, процент ошибок.
5. Построить сводные таблицы и графики для сравнительного анализа.
6. Сформулировать общие принципы стресс-тестирования.
7. Применить GigaCode с техниками zero-shot, one-shot, few-shot и задокументировать результаты.

---

## 3. Выбранные фреймворки

| Тип | Фреймворк | Характеристика |
|-----|-----------|----------------|
| Синхронный | Flask (WSGI) | Классический синхронный веб-фреймворк. Каждый запрос обрабатывается в отдельном потоке или процессе. GIL ограничивает параллелизм при выполнении CPU-задач. |
| Асинхронный | Sanic (asyncio) | Высокопроизводительный асинхронный веб-сервер на базе `asyncio`. Использует event loop для неблокирующей обработки тысяч соединений. |
| Асинхронный | aiohttp (asyncio) | Низкоуровневый асинхронный HTTP-фреймворк. Даёт полный контроль над event loop и маршрутизацией. |

**Исходный код:** `app_flask.py`, `app_sanic.py`, `app_aiohttp.py`, `locustfile.py`

## 4. Описание реализации

### 4.1 CPU-интенсивная задача

```python
CPU_N = 15_000_000  # 15 миллионов итераций

def heavy_cpu_sum():
    return sum(range(1, CPU_N + 1))
```

### 4.2 Эндпоинты

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| GET | `/cpu` | Блокирующее выполнение CPU-задачи (прямой вызов) |
| GET | `/cpu_fixed` | Неблокирующее выполнение (вынос в отдельный процесс или поток) |

### 4.3 Сравнение подходов

| Фреймворк | `/cpu` (блокирующий) | `/cpu_fixed` (неблокирующий) |
|-----------|----------------------|------------------------------|
| Flask | Прямой вызов в потоке, GIL блокирует другие потоки | `ProcessPoolExecutor` — выполнение в отдельном процессе |
| Sanic | Синхронный вызов блокирует event loop | `await loop.run_in_executor()` — вынос в поток |
| aiohttp | Синхронный вызов блокирует event loop | `await loop.run_in_executor(pool)` + `ProcessPoolExecutor` |

### 7.2 Анализ результатов

**При 50 пользователях (Flask и Sanic):**
- Ошибки в пределах 1–2% — серверы справляются с нагрузкой.
- RPS вырос до ~1.6 за счёт параллельной обработки.
- P95 резко вырос до 78–112 секунд — запросы выстраиваются в очередь.
- **Важное наблюдение:** Sanic на `/cpu` не показал ожидаемой полной блокировки event loop. При 50 пользователях зафиксировано лишь 2% ошибок. Вероятная причина — внутренняя многопоточная обвязка Sanic для синхронных вызовов.

**Ожидаемое поведение при 100+ пользователях:**
- Flask: рост ошибок и дальнейшее увеличение латентности.
- Sanic `/cpu`: резкий рост ошибок при исчерпании внутреннего пула потоков.
- Sanic/aiohttp `/cpu_fixed`: стабильная работа с минимальным процентом ошибок.

## 8. Применение GigaCode

### 8.1 Zero-shot — генерация эндпоинта Flask

**Промпт:**
> Напиши на Python эндпоинт для Flask, который суммирует числа от 1 до 15 миллионов и возвращает результат в JSON. Добавь замер времени.

**Техника:** Zero-shot (без примеров)

**Полученный код:**
```python
@app.route('/cpu')
def cpu_blocking():
    start = time.time()
    result = sum(range(1, 15000001))
    elapsed = time.time() - start
    return jsonify({'result': result, 'time': elapsed})
```

**Доработки:** Добавлен второй эндпоинт с `ProcessPoolExecutor`, вынесена функция `heavy_cpu_sum()`, добавлено округление времени.

**Оценка готовности:** 70% — код рабочий, но потребовал рефакторинга.

---

### 8.2 One-shot — генерация кода Sanic

**Промпт:**
> Вот пример эндпоинта на Flask:
> ```python
> @app.route('/cpu')
> def cpu_blocking():
>     result = heavy_cpu_sum()
>     return jsonify({'result': result, 'type': 'blocking'})
> ```
> Сделай такой же на Sanic с async/await. Для неблокирующего используй `loop.run_in_executor()`.

**Техника:** One-shot (один пример)

**Полученный код:**
```python
@app.get('/cpu')
async def cpu_blocking(request):
    result = heavy_cpu_sum()
    return json({'result': result, 'type': 'blocking'})

@app.get('/cpu_fixed')
async def cpu_nonblocking(request):
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(None, heavy_cpu_sum)
    return json({'result': result, 'type': 'nonblocking'})
```

**Доработки:** Минимальные — добавлен замер времени и импорт `asyncio`.

**Оценка готовности:** 90% — код практически готов к использованию.

## 9. Принципы стресс-тестирования

На основе проведённого эксперимента сформулированы следующие принципы:

1. **Тестировать оба режима работы**  
   Блокирующий (`/cpu`) и неблокирующий (`/cpu_fixed`) варианты демонстрируют принципиально разное поведение. Нельзя ограничиваться только оптимизированным сценарием.

2. **Наращивать нагрузку ступенчато**  
   Три точки (50, 100, 200 пользователей) позволяют оценить тренд деградации системы. Одна точка не даёт полной картины.

3. **Собирать перцентили, а не только среднее**  
   P95 и P99 выявляют проблемы, которые среднее значение скрывает. Например, медиана 37 секунд при P95 78 секунд указывает на сильный разброс времени ответа.

4. **Обеспечивать одинаковые условия сравнения**  
   Одинаковое железо, настройки (1 воркер/поток), фиксированная нагрузка (15 млн итераций). Без этого сравнение некорректно.

5. **Автоматизировать тесты**  
   Headless-режим + CSV-отчёты обеспечивают воспроизводимость. Ручное тестирование не даёт точных метрик.

6. **Анализировать ошибки, а не только успешные запросы**  
   2% ошибок при 50 пользователях — приемлемо. 97% ошибок при неправильной конфигурации — сигнал о проблеме.

7. **Искать причину ошибок, а не просто фиксировать**  
   Ошибки с кодом `0` при `threaded=False` и 200 пользователях привели к обнаружению причины (однопоточность) и её исправлению (`threaded=True`), что улучшило результаты.

8. **Учитывать архитектуру фреймворка**  
   Синхронный вызов в асинхронном обработчике ожидаемо блокирует event loop. Правильное решение — `run_in_executor()` или `ProcessPoolExecutor`.

## 10. Выводы

1. **Блокирующие CPU-операции значительно деградируют производительность** — при 50 пользователях латентность достигает 60–80 секунд даже при низком проценте ошибок.

2. **Вынос CPU-задач в отдельные процессы или потоки обязателен** — `/cpu_fixed` показывает сопоставимые или лучшие результаты по стабильности и предсказуемости.

3. **Асинхронные фреймворки не решают проблему CPU-нагрузки автоматически** — Sanic показал лишь 2% ошибок на `/cpu` вместо ожидаемых 100% благодаря внутренней многопоточности, но латентность осталась высокой.

4. **Выбор фреймворка зависит от типа нагрузки:**
   - **I/O-bound** (работа с БД, сеть) → Sanic / aiohttp
   - **CPU-bound** (вычисления) → Flask + ProcessPoolExecutor
   - **Смешанная нагрузка** → асинхронный фреймворк + `run_in_executor`

5. **GigaCode ускоряет разработку:**  
   - **Zero-shot** — для простых функций  
   - **One-shot** — для перевода между фреймворками  
   - **Few-shot** — для сложных сценариев  
   Экономия времени — от 50 до 80%.
