### 1. Регистрация и настройка окружения на платформе SourceCraft

**1.1. Регистрация в SourceCraft.dev**
Выполнен вход в систему SourceCraft.dev с использованием учетной записи Yandex. Аутентификация прошла успешно, доступ к платформе получен.

**1.2. Создание публичной организации**
Создана новая организация. В процессе создания был выбран тип видимости **"Публичная"** (Public), так как это обязательное условие для последующего хостинга статического сайта.

**1.3. Создание публичного репозитория**
В рамках созданной организации был создан репозиторий. Основные параметры репозитория:
*   **Название:** `my-static-site` (или указано ваше название).
*   **Видимость:** **Публичный** (Public). Это требование обусловлено политикой SourceCraft Sites, которая позволяет публиковать сайты только из публичных репозиториев.

**1.4. Генерация токенов и ключей доступа**
Для обеспечения безопасной работы с репозиторием по протоколам HTTPS и SSH были созданы следующие учетные данные:
*   **Персональный токен доступа (PAT):** Сгенерирован через веб-интерфейс SourceCraft. Токен использован для аутентификации при выполнении операций `git push` и `git pull` по HTTPS, а также как секрет для CI/CD-пайплайна.
*   **SSH-ключ:** Создана пара ключей (публичный и приватный). Публичный ключ был добавлен в настройки аккаунта SourceCraft, что позволило настроить подключение по SSH без ввода пароля. Файл конфигурации `~/.ssh/config` отредактирован для указания пользователя `git` и хоста `git.sourcecraft.dev`.

### 2. Подготовка репозитория и основа для статического сайта

**2.1. Клонирование репозитория**
Репозиторий был склонирован на локальную машину с использованием SSH (рекомендованный способ):
```bash
git clone git@git.sourcecraft.dev:<слаг_организации>/my-static-site.git
```

**2.2. Добавление основы для сайта на Hugo**
В корень репозитория был добавлен базовый шаблон для статического сайта, сгенерированный с помощью генератора **Hugo**. Структура включает:
*   Папку `content/` для Markdown-файлов.
*   Папку `layouts/` для HTML-шаблонов.
*   Папку `static/` для статических файлов (CSS, изображения).
*   Конфигурационный файл `config.toml`.
*   Временная папка `public/` (добавлена в `.gitignore`), в которую Hugo будет собирать готовый HTML-код.

**2.3. Настройка конфигурации для SourceCraft Sites**
В корне репозитория была создана служебная папка `.sourcecraft/`, содержащая файлы:
*   `sites.yaml` — конфигурация для публикации сайта. В параметре `root` указан путь к папке, в которую Hugo собирает сайт (по умолчанию `public/`), а в `ref` — ветка `main`, используемая для публикации.
*   `ci.yaml` — файл, описывающий CI/CD-процесс (подробнее в разделе 3).

### 3. Реализация CI/CD-процесса для Hugo

В рамках задания был разработан и развернут пайплайн непрерывной интеграции и доставки на базе встроенной CI/CD-системы SourceCraft.

**3.1. Структура пайплайна**
Пайплайн описан в файле `.sourcecraft/ci.yaml` и включает следующие этапы (джобы), выполняемые последовательно при каждом push в основную ветку `main`:

1.  **Подготовка окружения:**
    *   Использование образа `ubuntu:latest`.
    *   Установка необходимого ПО: `git`, `curl`, а также самого генератора Hugo (скачивание бинарного файла последней версии и добавление его в `PATH`).
2.  **Сборка сайта (`build`):**
    *   Запуск команды `hugo --minify` для генерации статических HTML-файлов из Markdown и шаблонов.
    *   Результат сборки сохраняется в папку `public/`, как указано в конфигурации.
3.  **Публикация артефактов (`deploy`):**
    *   Поскольку папка `public/` содержит конечный продукт, а SourceCraft Sites автоматически "подхватывает" изменения в указанной директории, отдельный шаг деплоя не требуется. Однако, для гарантии, в пайплайне был добавлен шаг, проверяющий наличие необходимых файлов (`index.html`) в `public/` после сборки.

**3.2. Расширение пайплайна: проверки синтаксиса Markdown**
В соответствии с дополнительным требованием задания, в пайплайн были интегрированы шаги по проверке качества исходного кода Markdown. Эти шаги выполняются **до** основного этапа сборки.

**Реализованные проверки:**
*   **Шаг 1: Проверка ссылок (`markdown-link-check`).**
    *   Установлен и запущен утилита `markdown-link-check`.
    *   Команда: `npx markdown-link-check ./content/**/*.md --config .linkcheck.json`.
    *   Проверяет: Наличие битых (неработающих) внутренних и внешних гиперссылок. Конфигурационный файл `.linkcheck.json` настроен на игнорирование некоторых внешних ресурсов (например, `localhost`) и не строгую проверку HTTPS.
*   **Шаг 2: Проверка форматирования и орфографии (`markdownlint`).**
    *   Установлен и запущен инструмент `markdownlint-cli2`.
    *   Команда: `npx markdownlint-cli2 "content/**/*.md"`.
    *   Проверяет: Соблюдение правил оформления Markdown (например, длина строки, наличие пробелов после заголовков, правильность вложенности списков). Файл конфигурации `.markdownlint.json` позволяет гибко настраивать правила.

**Важно:** Оба шага проверок интегрированы в пайплайн с жестким условием — при обнаружении ошибок выполнение пайплайна прерывается, и **сборка сайта не происходит**. Это предотвращает публикацию некачественного или некорректного контента.
