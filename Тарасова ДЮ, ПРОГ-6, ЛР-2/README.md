# Лабораторная работа №2: Методы оптимизации вычислений (потоки, процессы, Cython, GIL)

## Цель работы
Исследовать методы ускорения вычислительных задач в Python на примере численного интегрирования методом левых прямоугольников. Сравнить эффективность:
- многопоточности (ThreadPoolExecutor),
- многопроцессности (ProcessPoolExecutor),
- компиляции с помощью Cython (с типами и без GIL),
- параллелизма с освобождением GIL (nogil).

## Реализованные методы

| Итерация | Название                     | Файл                          | Особенность                                   |
|----------|------------------------------|-------------------------------|-----------------------------------------------|
| 1        | Базовый Python               | `integration.py`              | Последовательный цикл, вызов Python-функции  |
| 2        | Потоки (ThreadPool)          | `integration_threads.py`      | Разбиение на потоки, нет ускорения из-за GIL |
| 3        | Процессы (ProcessPool)       | `integration_processes.py`    | Обход GIL, ускорение на многоядерных системах|
| 4        | Cython (cdef, C-функции)     | `integration_cython.pyx`      | Типизация, вызов libc.math.sin без GIL       |
| 5        | Cython + nogil + prange      | `integration_cython.pyx`      | Параллельный цикл prange с освобождением GIL |

## Результаты замеров (пример для n_iter=10⁷, cos на [0, π/2], 4 воркера)

| Метод                         | Время (сек) | Ускорение (отн. Python) |
|-------------------------------|-------------|--------------------------|
| Python (базовый)              | 1.452       | 1.00                     |
| Потоки (ThreadPool)           | 1.483       | 0.98                     |
| Процессы (ProcessPool)        | 0.381       | 3.81                     |
| Cython (sin, один поток)      | 0.082       | 17.7                     |
| Cython + nogil + prange       | 0.024       | 60.5                     |

*Измерения проводились на Intel Core i7-8700K (6 ядер), Python 3.11.*

## Анализ и выводы

1. **Потоки (Iteration 2)**  
   – Не дают ускорения для CPU-задач на чистом Python из-за GIL.  
   – Даже небольшие накладные расходы на создание потоков ухудшают время.

2. **Процессы (Iteration 3)**  
   – Работают независимо, каждый со своим GIL.  
   – Ускорение близко к количеству ядер (при 4 процессах → 3.8×).  
   – Недостаток: высокие накладные расходы на передачу данных и запуск.

3. **Cython с C-типами (Iteration 4)**  
   – Объявление переменных как `cdef double` и использование C-функции (`libc.math.sin`) ускоряет код в ~18 раз без распараллеливания.  
   – Исчезают накладные расходы на вызов Python-функций в цикле.

4. **Cython + nogil + prange (Iteration 5)**  
   – Освобождение GIL и параллельный цикл `prange` даёт ещё ~3× ускорение сверх однопоточного Cython.  
   – Итоговое ускорение до **60×** по сравнению с чистым Python.  
   – Это самый эффективный метод для численных расчётов на CPU.

## Применение примитивов синхронизации

В данной задаче **нет необходимости** в мьютексах или семафорах, потому что:
- Каждый воркер (поток/процесс) работает с независимыми данными (свой подынтервал, локальный аккумулятор).
- Результаты собираются после завершения всех воркеров.
- Разделяемые изменяемые переменные отсутствуют.

Если бы мы использовали общий аккумулятор без синхронизации, потребовалась бы блокировка (например, `threading.Lock` или `cython.parallel.parallel` с `critical`), но это привело бы к сериализации и потере производительности.

## Выводы по Python 3.14 (опциональный GIL)

В версии Python 3.14+ ожидается возможность запуска интерпретатора без GIL (`--disable-gil`). Это позволит истинно параллельно выполнять потоки с обычным Python-кодом. В такой среде:
- **Итерация 2** (обычные потоки) даст ускорение, сравнимое с процессами.
- Отпадёт необходимость в `nogil` для Cython – потоки будут работать параллельно автоматически.
- Однако потребуется перепроверка библиотек на thread-safety.

## Итоговое заключение

- Для CPU‑интенсивных задач на чистом Python **потоки бесполезны** – используйте процессы или компилируемые инструменты.
- **Cython** с правильной типизацией и освобождением GIL даёт наилучшую производительность (до 60× ускорения).
- **Numba** (например, с `@njit(parallel=True)`) – альтернатива, более простая в использовании, но предоставляет меньше контроля.
- В реальных проектах выбор метода зависит от сложности вычислений, требований к переносимости и готовности использовать компиляцию.
